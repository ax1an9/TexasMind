# Poker-Agent 集成 - 架构对比与选型指南

---

## 集成方案对比

### 方案 A: 直接集成（Current State）

**描述**: poker-agent 独立运行，Java 侧使用 SimpleHoldemAgent

| 指标 | 评分 | 说明 |
|------|------|------|
| 实现复杂度 | ⭐ | 无需改动，已完成 |
| 部署复杂度 | ⭐ | 两个独立进程 |
| AI 智能度 | ⭐⭐ | SimpleHoldemAgent 较弱 |
| 扩展性 | ⭐⭐ | 难以切换 agent |
| 用户体验 | ⭐⭐ | 本地 AI 对手较弱 |

**适用场景**: 
- 原型验证
- 简单演示
- 不需要智能 AI 的场景

---

### 方案 B: gRPC 实时决策（**推荐**）

**描述**: Java 通过 gRPC 远程调用 Python agent，本地 agent 作为降级

| 指标 | 评分 | 说明 |
|------|------|------|
| 实现复杂度 | ⭐⭐⭐ | 需要 3 个核心类 + adapter |
| 部署复杂度 | ⭐⭐⭐ | 需启动 2 个服务 + gRPC |
| AI 智能度 | ⭐⭐⭐⭐⭐ | 使用 LLM 推理 |
| 扩展性 | ⭐⭐⭐⭐ | 易于替换/升级 agent |
| 用户体验 | ⭐⭐⭐⭐⭐ | 对手更强 |
| 容错性 | ⭐⭐⭐⭐ | 自动降级到本地 |

**适用场景**: 
- 生产环境
- 需要强 AI 对手
- 多模型实验
- 长期维护

---

### 方案 C: 批量决策（Offline）

**描述**: 使用对局回放数据 fine-tune agent，离线生成策略表

| 指标 | 评分 | 说明 |
|------|------|------|
| 实现复杂度 | ⭐⭐⭐⭐ | 需要数据管道 + 模型训练 |
| 部署复杂度 | ⭐⭐ | 导入策略表即可 |
| AI 智能度 | ⭐⭐⭐⭐ | 根据数据而定 |
| 决策延迟 | ⭐ | < 1ms |
| 可维护性 | ⭐⭐ | 难以快速迭代 |

**适用场景**: 
- 策略已收敛
- 对延迟要求极高
- 模型不再频繁变更

---

### 方案 D: 混合模式（HYBRID）

**描述**: 根据情况切换 gRPC 或本地 agent

| 指标 | 评分 | 说明 |
|------|------|------|
| AI 智能度 | ⭐⭐⭐⭐⭐ | 最优情况下等同方案 B |
| 容错性 | ⭐⭐⭐⭐⭐ | Python server 离线也能玩 |
| 成本效率 | ⭐⭐⭐ | 避免不必要的 gRPC 调用 |
| 可观测性 | ⭐⭐⭐⭐ | 可追踪每次决策的来源 |

**切换策略示例**:
```
- 前 5 轮：本地（热身）
- 中间：根据 Python server 可用性切换
- 后期：优先 gRPC（需要更强决策）
```

## 选型建议

### 短期（第一周）

✅ **采用方案 B (gRPC 实时决策)**

**原因**:
1. 投入回报率最高（核心改动 ~500 行代码）
2. 性能可接受（延迟 < 2s）
3. 天然支持降级（容错好）
4. 便于后续扩展

### 中期（第二周至第二个月）

📊 **开始收集对局数据，为方案 C 准备**

**步骤**:
1. 记录所有对局（已有 ReplayRecorder）
2. 定期导出标准化数据集
3. 分析 agent 决策质量（胜率、AUM 等）

### 长期（2-3 个月后）

🚀 **根据数据评估升级方案 C 或 D**

**决策树**:
```
if (agent_winrate > 0.65 and model_stable):
    → 迁移到 Offline 策略表 (方案 C)
else if (grpc_error_rate > 10%):
    → 启用 HYBRID 混合模式 (方案 D)
else:
    → 继续优化 gRPC 性能 (方案 B)
```

## 关键指标定义

### 方案 B 性能指标

```yaml
决策延迟:
  P50: < 500ms
  P95: < 1500ms
  P99: < 2500ms

可用性:
  Python server 响应率: >= 95%
  自动降级触发率: <= 5%
  降级后胜率变化: -2% ~ -5%

用户体验:
  AI 对手胜率: 55% ~ 70%
  用户体验评分: >= 4/5
```

### 迁移到方案 C 的条件

```yaml
数据准备:
  ✓ 对局数量: >= 10000
  ✓ 覆盖度: 所有 position/hand rank 组合
  ✓ 对手多样性: >= 5 个不同 agent/玩家

模型质量:
  ✓ 训练集胜率: > 65%
  ✓ 验证集胜率: > 63%
  ✓ 过拟合差: < 2%
  ✓ 推理延迟: < 1ms

运维就绪:
  ✓ 策略版本管理
  ✓ A/B 测试框架
  ✓ 快速回滚机制
```

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解方案 |
|------|------|------|--------|
| Python server 宕机 | 中 | 高 | 自动降级 + 健康检查 |
| gRPC 延迟过高 | 低 | 中 | 增加超时，优化模型 |
| Proto 版本不匹配 | 低 | 高 | 版本兼容性测试 |
| LLM API 成本过高 | 中 | 中 | 缓存、批处理、本地模型 |
| 并发 agent 过多 | 低 | 中 | 连接池、资源限制 |

## 总结

**建议**: 
- ✅ **立即采用方案 B (gRPC 实时决策)**
- 📋 **并行收集对局数据**
- 🚀 **2-3 个月后评估升级到方案 C 或 D**

**关键成功因素**:
1. 自动降级机制（确保容错）

3. 灰度上线（逐步扩大范围）
4. 数据驱动迭代（基于实际指标优化）
